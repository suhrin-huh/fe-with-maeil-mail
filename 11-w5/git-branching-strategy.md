# 알고 계신 git 브랜치 전략들에 대해 소개해주세요.

Git 브랜치 전략은 협업의 효율성과 배포 안정성을 좌우하는 핵심 요소이다. 대표적인 전략으로는 Git Flow, GitHub Flow, Trunck-Based Development가 있다.
각 전략은 프로젝트 규모, 배포 주기, QA 방식에 따라 장단점이 나뉜다.

## Git Flow

Git Flow는 대규모 프로젝트나 릴리스 주기가 긴 서비스에서 많이 사용되는 전략으로, 브랜치 구조가 체계적으로 나뉘어 있다.

### 주요 브랜츠

- main : 실제 운영(프로덕션) 코드
- develop : 다음 릴리스에 포함될 작업이 모이는 브랜치
- feature/\* : 기능 개발용 브랜치
- release/\* : 릴리스 준비용 브랜치
- hotfix/\* : 긴급 수정용 브랜치

기능 개발은 feature 브랜치에서 이루어지며, 완료된 후 develop 브랜치에 병합된다. 이후 release 브랜치를 따로 만들어 QA와 최종 검증을 거친후에 프로덕션 코드를 관리하는 main에 병합한다. 긴급한 수정사항이 발생할 경우 hotfix 브랜치를 만들어 신속히 배포하고, 수정사항을 develop에도 반영한다.

### 장단점

- 장점 : 명확한 역할 분리가 이루어져 있어 안정성 보장에 유리하다. QA 과정과 릴리스 관리가 체계적이다.

- 단점 : 브랜치 구조가 복잡해 작업 흐름이 무거울 수 있다. 빠른 배포 환경에서는 비효율적이다.

=> 금융, 공공 서비스 등 높은 안정성과 품질 관리가 필요한 프로젝트에 주로 사용되는 전략이다.

## GitHub Flow

GitHub Flow는 Git Flow의 단순 버전으로, 빠른 배포와 지속적인 변경 반영이 필요한 환경에서 사용된다.

### 주요 브랜치

모든 기능은 feature 브랜치에서 개발되어 코드 리뷰 후 바로 main에 병합된다. 병합 즉시 배포하는 경우가 많다.

### 장단점

- 장점 : 구조가 단순하여 소규모 팀, 스타트업에서 생산성이 극대화된다. 배포 주기가 매우 짧은 환경에 최적화되어 있다.
- 단점 : 별도의 QA, release 브랜치가 없어 품질 관리 부담이 증가한다.

=> 스타트업, 빠른 테스트 & 배포가 중요한 제품에 주로 사용된다.

## Trunk-Based Development

Trunk-Based Development(TBD)는 가장 단순하고 극단적으로 빠른 배포를 목표로 하는 전략이다.

### 주요 브랜치

단일 브랜치(main 또는 trunk)중심이다. 작업을 main 브랜치에 직접 커밋하거나, feature 브랜치를 만들더라도 짧은 기간 내 병합된다. 지속적 통합(CI)과 자동화 테스트 환경이 필수이다.

### 장단점

- 장점 : 병합 충돌이 크게 줄어들며, DevOps, CI/CD 환경과 높은 궁합을 이룬다.
- 단점 : 자동화 테스트 및 품질 통제 시스템이 미흡하면 안정성이 저하될 수 있다.

=> DevOps 기반 또는 고도화된 자동화 배포 환경에 적합하다.
